# Decision Log (MVP)

Этот файл фиксирует ключевые архитектурные и продуктовые решения для MVP,
чтобы было проще объяснить, почему проект реализован именно так.

## 1) Почему FastAPI

**Решение:** использовать FastAPI как базовый веб-фреймворк.

**Почему:**
- Быстрый старт для MVP: минимальный boilerplate.
- Встроенная валидация запросов и удобные типы Python.
- Автоматическая документация OpenAPI (`/docs`) полезна для проверки API-контракта.

**Цена решения / компромисс:**
- Нужно внимательно следить за зависимостями экосистемы (`httpx`, `python-multipart` для тестов/форм).

## 2) Почему SQLite для MVP

**Решение:** хранить данные в SQLite на этапе MVP.

**Почему:**
- Простой локальный запуск без отдельного сервера БД.
- Достаточно для сценария «один инстанс / небольшой объем».
- Хорошо подходит для быстрого прототипирования импорта и фильтрации событий.

**Цена решения / компромисс:**
- Ограничения по concurrency и масштабируемости по сравнению с PostgreSQL.

## 3) Почему строгий CSV-контракт

**Решение:** проверять header CSV строго и в точном порядке:
`timestamp,source,level,message,metric_value,tag`.

**Почему:**
- Снижает неоднозначность в импорте.
- Упрощает диагностику ошибок у пользователя.
- Соответствует `docs/data-format.md` и `docs/tz.md`.

**Цена решения / компромисс:**
- Менее гибко к «почти правильным» CSV-файлам.

## 4) Почему разделение routers -> services

**Решение:** вынести логику из `app/main.py` в отдельные модули `routers/` и `services/`.

**Почему:**
- Эндпоинты становятся тонкими (только прием параметров/ответ).
- Бизнес-логика переиспользуется и проще тестируется.
- Соответствует целевой архитектуре из `docs/architecture.md`.

**Цена решения / компромисс:**
- Файлов больше, чем в монолите, нужно привыкнуть к структуре.

## 5) Почему единый builder фильтров для /events и export

**Решение:** использовать общий `build_event_filters(...)`.

**Почему:**
- Синхронизирует поведение `/events` и `/events/export.csv`.
- Уменьшает риск расхождений (один endpoint фильтрует иначе, чем другой).
- Облегчает поддержку и изменения фильтров в одном месте.

**Цена решения / компромисс:**
- Нужна аккуратная обработка ошибок (например, bad timestamp) в одном общем слое.

## 6) Почему ISO-формат timestamp в ответах

**Решение:** сериализовать время через `isoformat()`.

**Почему:**
- Формат предсказуем и удобен для клиентского кода.
- Стабильный контракт между API и UI/скриптами.

**Цена решения / компромисс:**
- Нужно единообразно применять во всех endpoint-ответах.

## 7) Почему тесты покрывают и success, и error paths

**Решение:** добавить интеграционные тесты на валидные и невалидные сценарии.

**Почему:**
- Проверяется не только «счастливый путь», но и реальное поведение при ошибках данных.
- Поддерживает acceptance из `docs/testplan.md`.

**Цена решения / компромисс:**
- Больше времени на поддержку тестов при изменениях API-контракта.

## 8) Почему в тестах используется in-memory SQLite

**Решение:** в `tests/conftest.py` поднимать отдельную in-memory БД (`sqlite://`).

**Почему:**
- Тесты изолированы от рабочей `app.db`.
- Быстрый прогон и предсказуемое состояние данных.
- Удобно для CI и локальной проверки без ручной очистки файлов.

**Цена решения / компромисс:**
- Нужно отдельно проверять поведение на файловой SQLite/другой БД, если появятся особенности.

## 9) Почему добавлены `httpx` и `python-multipart`

**Решение:** установить дополнительные зависимости в dev-окружение.

**Почему:**
- `httpx` нужен для `fastapi.testclient`.
- `python-multipart` нужен для endpoint-ов c `UploadFile`/form-data (`POST /import`).

**Цена решения / компромисс:**
- Чуть больше зависимостей для сопровождения, но без них тесты и upload-сценарий ломаются.

## 10) Почему ImportRun history вынесен в отдельный endpoint

**Решение:** добавить `GET /imports` вместо включения истории импортов в `/dashboard`.

**Почему:**
- История импортов — отдельный сценарий использования (операционный журнал).
- У endpoint есть собственная пагинация (`limit/offset`).
- Dashboard остается легким и сфокусированным на агрегатах событий.

**Цена решения / компромисс:**
- Увеличивается число endpoint-ов, нужно поддерживать дополнительный API-контракт.

## 11) Почему top-sources сделан отдельным endpoint

**Решение:** добавить `GET /dashboard/top-sources` с двумя блоками: `ERROR` и `WARN`.

**Почему:**
- Это логичное расширение v0.2 без изменения контракта базового `/dashboard`.
- Можно отдельно контролировать `limit` для top-выборок.
- Удобно для будущего UI-виджета «проблемные источники».

**Цена решения / компромисс:**
- Появляется дополнительный запрос для UI, если нужно одновременно показать и сводку, и top-sources.

## 12) Почему UI сделан server-side на Jinja

**Решение:** добавить простой server-rendered UI (`/ui/*`) вместо SPA.

**Почему:**
- Быстрый способ закрыть v0.2 без усложнения фронтенд-стека.
- UI прямо отражает текущий API и легче дебажится.
- Хорошо подходит для внутреннего инструмента и MVP-подхода.

**Цена решения / компромисс:**
- Меньше интерактивности по сравнению со SPA.
- Для сложного UX в будущем, вероятно, нужен более богатый фронтенд-слой.

## 13) Почему в UI даты выбраны через datetime-local

**Решение:** использовать `input type="datetime-local"` в `/ui/events`.

**Почему:**
- Пользователю проще выбрать диапазон времени кликом, чем вводить ISO вручную.
- Снижает число ошибок в фильтрах start/end.

**Цена решения / компромисс:**
- Нужна нормализация локального времени в timezone-aware формат перед запросом к сервисному слою.

## 14) Почему таблица events обновляется через HTMX partial

**Решение:** добавить endpoint `GET /ui/events/table` и подключить HTMX на форме фильтров.

**Почему:**
- Пользователь быстрее работает с фильтрами (без полной перезагрузки страницы).
- Бэкенд остается простым: отдается HTML-фрагмент из Jinja.
- Полный fallback сохраняется: без JS форма отправляется на `GET /ui/events`.

**Цена решения / компромисс:**
- Нужно поддерживать два рендера: полная страница (`/ui/events`) и partial (`/ui/events/table`).
