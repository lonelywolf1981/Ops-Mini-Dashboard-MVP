# ТЗ: Ops Mini Dashboard MVP

## 1. Цель
Создать внутренний веб-инструмент для импорта и анализа событий/логов из CSV:
- обеспечить прозрачность (сводка по уровням, источникам, времени),
- упростить поиск причин (фильтры, быстрый просмотр истории),
- дать возможность выгрузки данных для дальнейшего анализа.

## 2. Пользователи
- Инженер / лаборант: смотрит события эксперимента, ошибки записи, отклонения.
- Техслужба / эксплуатация: мониторит типовые проблемы, ищет первопричины.
- IT/DevOps: смотрит сетевые/сервисные события в виде простого журнала.

## 3. Термины
- **Событие (Event)** — запись вида timestamp/source/level/message (+ опциональные metric_value/tag).
- **Импорт (ImportRun)** — один запуск загрузки файла, с итоговой статистикой.

## 4. Область MVP
### Входит
- Импорт CSV через веб-форму.
- Хранение в БД (MVP: SQLite).
- Dashboard: счётчики + графики + последние события.
- Events: таблица событий с фильтрами и пагинацией.
- Экспорт текущей выборки событий в CSV.

### Не входит
- Авторизация/роли.
- Реальное подключение к оборудованию.
- Реалтайм (стриминг).
- Сложная аналитика/аномалии.
- Алертинг/уведомления.
- Многопользовательские сценарии и ограничения доступа.

## 5. Источник данных и формат
Файл CSV в UTF-8, разделитель — запятая, заголовок обязателен.
Полный формат: см. `data-format.md`.

Минимальные поля:
- timestamp (ISO 8601)
- source (string)
- level (INFO/WARN/ERROR)
- message (string)
Опционально:
- metric_value (float)
- tag (string)

## 6. Функциональные требования

### 6.1. Импорт
**Форма**: загрузка `.csv`.

**Поведение:**
- Проверка расширения файла `.csv`.
- Проверка заголовка: строго фиксированный список колонок.
- Строки с пустыми `source/level/message` пропускать (skipped).
- `level` должен быть одним из `INFO|WARN|ERROR`, иначе skipped.
- Ошибка парсинга timestamp/metric_value → errors (строка не добавляется).
- В конце — статистика импорта:
  - inserted, skipped, errors, import_run_id.

**Хранение:**
- Каждое событие должно ссылаться на `import_run_id`.

### 6.2. Просмотр списка событий (Events)
Экран/эндпоинт: список событий, сортировка по времени (desc).

Фильтры:
- level (равно)
- source (равно)
- q (подстрока в message)
- start/end (диапазон времени)
Параметры:
- limit (1..500), offset (>=0)

### 6.3. Dashboard
Показывать:
- Total events
- Counts by level
- Unique sources
- График событий по времени (агрегация по дням/часам — допустимо по дням)
- Последние 20 событий

### 6.4. Экспорт CSV
- Экспортирует текущую выборку по тем же фильтрам, что и Events.
- Ограничение MVP: максимум 5000 строк.

## 7. Нефункциональные требования
- Простота развёртывания (локально: одна команда запуска).
- Предсказуемость: понятные ошибки при неверном формате.
- Кодовая структура: router/service/model (без монолита в одном файле).
- Документация: минимум README + decision log + defense.

## 8. Ограничения и допущения
- БД SQLite в MVP, миграции необязательны.
- Один пользователь/одна инстанция — нормальный сценарий для MVP.
- UI может быть простым, главное — полезность.

## 9. Критерии приёмки (Acceptance Criteria)
MVP считается готовым, если:
1) Проект запускается по инструкции из README.
2) Импорт `sample_events.csv` выполняется без ошибок формата.
3) `GET /events` возвращает данные, фильтры работают.
4) Dashboard показывает корректные счётчики (total/by_level/unique_sources).
5) Экспорт CSV работает и соответствует выбранным фильтрам.
6) В репозитории присутствуют документы:
   - `docs/TZ.md`, `docs/architecture.md`, `docs/api.md`,
     `docs/data-format.md`, `docs/test-plan.md`,
     `docs/decision-log.md`, `docs/project-defense.md`.

## 10. План работ (минимальный)
- Этап 1: каркас приложения + БД + модели.
- Этап 2: импорт.
- Этап 3: events + фильтры + экспорт.
- Этап 4: dashboard.
- Этап 5: упаковка и документация.
